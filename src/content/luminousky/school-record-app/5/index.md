## DB 전체 암호화 vs 필드 단위 암호화

암호화를 도입하기로 결정했을 때 선택지는 두 가지였다.

**SQLCipher** (DB 전체 암호화): DB 파일 자체를 암호화해서 비밀번호 없이는 파일을 열 수 없다. 구현이 단순하지만 Tauri 환경에서 rusqlite와 통합하려면 추가 빌드 설정이 필요하고, 암호화 여부를 나중에 토글하기가 어렵다.

**필드 단위 암호화**: 민감한 컬럼(`Student.name`, `ActivityRecord.content`, `ActivityRecordHistory.content`)만 암호화 텍스트로 저장한다. DB 구조 자체는 평문이지만, 실제 데이터는 읽을 수 없다. 암호화를 비활성화하면 기존 컬럼을 그대로 복호화해서 평문으로 돌릴 수 있다.

사용자가 중간에 암호화를 켜고 끄거나 비밀번호를 변경할 수 있어야 한다는 요구사항 때문에 **필드 단위 암호화**를 선택했다.

## 설계

### 저장 형식

```
nonce_b64:ciphertext_b64
```

AES-256-GCM은 nonce를 매 암호화마다 새로 생성해야 한다. nonce와 암호문을 콜론으로 구분해 단일 TEXT 컬럼에 저장한다.

### 키 파생

비밀번호 → PBKDF2-HMAC-SHA256 → 32바이트 키

- 반복 횟수: **600,000회** (OWASP 2026 권고값, 초기 구현은 100,000회였다가 상향)
- salt: 16바이트 랜덤, `APP_CONFIGS` 테이블에 저장
- 파생 키는 메모리(`CryptoState`)에만 유지 — DB에 저장하지 않는다

`CryptoState`에 `ZeroizeOnDrop`을 구현해 메모리에서 키를 안전하게 소거한다.

### 비밀번호 검증

`"school-record-verify"` 고정 문자열의 암호화본을 `APP_CONFIGS`에 저장해둔다. 비밀번호를 입력하면 이 값을 복호화해서 검증한다. 비밀번호가 틀리면 복호화가 실패하거나 결과가 일치하지 않는다.

### 원자성

암호화 활성화/비활성화/비밀번호 변경은 모두 단일 SQLite 트랜잭션으로 처리한다. 작업 시작 전에 자동 백업 파일을 생성하고, 백업 실패 시 작업을 중단한다.

## 발견된 버그들

검증 과정에서 6개의 버그를 수정했다.

**빈 문자열 처리 버그** — 가장 심각했다. `transform_all_data`에서 빈 content를 그대로 암호화하면 `":"`처럼 nonce만 있는 형태가 저장됐다. 나중에 복호화를 시도하면 파싱 오류가 발생한다. `skip_empty: true` 옵션으로 빈 문자열을 passthrough하도록 수정했다.

**중복 활성화 체크 순서** — PBKDF2 600,000회 연산을 하고 난 다음에 "이미 활성화됨" 체크를 했다. 순서를 바꿔서 불필요한 연산을 방지했다.

**비밀번호 변경 시 백업 누락** — `enable_encryption_impl`에는 백업이 있었지만 `change_encryption_password_impl`에는 없었다.

**PasswordModal 취소 버튼** — unlock 모드에서 취소 버튼이 없어서 모달을 닫을 방법이 없었다.

## 설계상 한계 (의도적 결정)

| 항목 | 결정 |
|------|------|
| 비밀번호 분실 시 복구 불가 | 설계 결정 — 사용자에게 명시적 경고 표시 |
| salt·verify token이 DB에 평문 저장 | OS keychain 미사용으로 인한 한계, 위험도 낮음 |

salt가 평문이어도 안전한 이유는, PBKDF2의 600,000회 반복이 brute force를 실질적으로 불가능하게 만들기 때문이다.

## 결과

최종 테스트 232개 전체 통과. 백엔드 암호화 코드는 `src-tauri/src/crypto.rs`, 커맨드는 `src-tauri/src/commands/crypto.rs`에 분리됐다.

이 기능을 검증하면서 테스트의 가치를 다시 확인했다. 버그 6개 중 3개는 코드를 읽는 것만으로는 발견하기 어려웠고, 실제 암호화/복호화 흐름을 테스트에서 직접 돌려봐야 드러났다.
