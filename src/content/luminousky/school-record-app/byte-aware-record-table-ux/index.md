## 도메인 제약이 UI를 결정한다

NEIS(교육행정정보시스템)는 학생부 기록 텍스트에 바이트 한도가 있다. 학생 한 명당, 영역별로 입력할 수 있는 바이트 수가 정해져 있다. 이 제약이 RecordSection 전체 UI 방향을 결정했다.

단순히 텍스트를 입력하고 저장하는 기능이 아니었다. 교사가 각 학생의 남은 바이트를 실시간으로 보면서 작성하고, 초과한 학생을 한눈에 파악할 수 있어야 했다.

## 바이트 계산 — TextEncoder 싱글턴

한글은 UTF-8에서 3바이트다. JavaScript의 `string.length`는 문자 수를 반환하므로 바이트 계산에는 `TextEncoder`를 써야 한다.

초기 구현은 계산할 때마다 `new TextEncoder()`를 만들었다. 모든 셀 입력마다 인스턴스를 반복 생성하는 방식이었다. 이것을 모듈 상단에 싱글턴으로 꺼냈다.

```js
const encoder = new TextEncoder();

function byteLength(str) {
  return encoder.encode(str).byteLength;
}
```

`TextEncoder`는 상태가 없다. 매번 새로 만들 이유가 없다. 실측 성능 차이보다 "불필요한 반복 할당을 만들지 않는다"는 원칙이 중요했다.

## reactive(Map)과 ref(Map)의 차이

셀 내용을 Map으로 관리했다. 처음에는 `ref(Map)`으로 선언했다.

```js
const cellContent = ref(new Map());
```

문제는 `ref`가 Map 전체를 하나의 값으로 추적한다는 것이다. 특정 키만 바꿔도 의존하는 모든 computed가 재실행된다. 학생이 30명, 활동이 10개면 테이블이 300개 셀의 변화를 모두 처리한다.

`reactive(Map)`으로 바꾸면 Vue가 Map 내부 키 단위로 추적한다.

```js
const cellContent = reactive(new Map());
```

이제 특정 학생·활동 조합의 셀만 변경하면 그 키에 의존하는 computed만 재실행된다. 입력 시 반응성이 눈에 띄게 개선됐다.

## sticky 열 투명 문제

RecordSection은 학생 이름과 바이트 합계 열을 `position: sticky`로 고정한다. 열이 고정되면 콘텐츠를 스크롤해도 제자리에 남는다.

그런데 배경이 투명하면 스크롤한 내용이 sticky 열 뒤로 비친다. 테이블을 가로로 스크롤할 때 활동 텍스트가 이름 열 아래로 드러났다.

해결은 단순하다. sticky 열에 배경색을 명시적으로 지정하면 된다.

```css
.sticky-cell {
  position: sticky;
  left: 0;
  background-color: var(--color-surface);
  z-index: 2;
}
```

기본 배경이 `transparent`인 것이 함정이다. sticky 열은 반드시 불투명 배경을 가져야 한다.

행 강조 같은 상태 색상을 sticky 열에 적용할 때도 같은 함정이 있다. 강조 배경을 반투명(`rgba`)으로 쓰면 뒷 내용이 비친다. 강조 상태에서는 불투명 색상으로 덮어써야 한다.

## 셀 높이 동기화

textarea 셀은 내용에 따라 자동으로 높이가 늘어난다. 그런데 같은 행의 바이트 열, 이름 열, 미리보기 열은 각자 독립적으로 높이가 정해지므로 행 높이가 들쭉날쭉해졌다.

`syncRowHeights()` 함수를 만들어 각 행에서 모든 셀의 높이를 측정하고, 최댓값으로 맞춘다.

```js
function syncRowHeights() {
  rows.forEach((_, rowIdx) => {
    const cells = tableEl.querySelectorAll(`[data-row="${rowIdx}"]`);
    const maxH = Math.max(...[...cells].map(c => c.scrollHeight));
    cells.forEach(c => (c.style.height = `${maxH}px`));
  });
}
```

이 함수를 호출해야 하는 시점이 여럿이었다: 셀 내용 변경 시, 글자 크기 변경 시, 미리보기 열 ON/OFF 시, 창 리사이즈 시. 이벤트 발생마다 debounce를 걸어서 과도한 재계산을 막았다.

미리보기 열을 추가할 때 `syncRowHeights`가 미리보기 열 높이를 포함하지 않는 버그가 있었다. 새 열이 추가될 때마다 셀렉터에 포함시켜야 한다는 것을 잊기 쉽다.

## 미리보기 열

기록을 작성하면서 "실제로 내보낼 때 어떻게 보일지"를 바로 확인하고 싶다는 요구가 있었다. 치환 규칙이 적용된 결과, 유의어 검사 하이라이트, 최종 출력 형태를 미리 볼 수 있으면 작성 중에 실수를 줄일 수 있다.

미리보기 열은 바이트 열 왼쪽에 배치하고 틀고정에 포함시켰다. 활동마다 다른 형광펜 색상을 부여해서 어느 활동의 기록인지 구분할 수 있게 했다.

미리보기 열의 left 오프셋은 다른 sticky 열들의 너비 합에 따라 동적으로 계산해야 했다. sticky 열이 여러 개 있을 때 각각의 `left` 값을 누적해서 지정하지 않으면 열들이 겹친다.

```css
/* 이름 열: left 0 */
/* 바이트 열: left = 이름열 너비 */
/* 미리보기 열: left = 이름열 너비 + 바이트열 너비 */
```

## 자동저장 피드백

셀을 수정하면 1초 debounce 후 자동으로 저장된다. 사용자가 저장 버튼을 누르지 않아도 된다.

저장 상태 피드백의 첫 구현은 셀 테두리 색상 변경이었다 — 저장 중: 파란 테두리, 저장 완료: 원래 색. 이후 배경색으로 바꿨다. 테두리는 너무 미묘해서 인지하기 어렵고, 배경색 변화가 더 눈에 잘 띈다.

저장 실패 시에는 빨간 테두리를 3초 표시한다. 성공과 실패를 다른 피드백 채널로 구분하는 것이 좋다 — 성공은 조용히, 실패는 명확하게.

## Copy 버튼과 텍스트 정규화

바이트 열에 학생 전체 기록을 클립보드에 복사하는 버튼을 추가했다. NEIS에 붙여넣을 때 사용한다.

복사할 때 줄바꿈을 공백으로 바꾸고, 연속된 공백을 정규화한다. NEIS가 줄바꿈을 처리하지 못하거나 다르게 해석하는 경우가 있기 때문이다. "Copied!" 텍스트를 1초 표시하고 원래 상태로 돌아온다.

클릭 후 버튼 텍스트가 바뀌는 패턴은 `markCopied` 헬퍼로 추출했다. 여러 Copy 버튼에서 같은 로직을 반복하지 않도록.

## 인적사항 열 숨기기

학년, 반, 번호 열은 학생을 구분하는 용도다. 열이 많아지면 작업 공간이 좁아지므로 이 열을 토글로 숨길 수 있게 했다. 기록 작업에 집중할 때는 이름만 보이면 충분하다.

숨겼을 때 sticky 열의 left 오프셋이 다시 재계산되어야 했다. 숨긴 열의 너비만큼 오프셋이 줄어든다.

## 활동 열 접힘

활동 열이 많으면 가로 스크롤이 길어진다. 각 활동 열을 접어서 너비를 최소화하는 기능을 추가했다.

접힌 열은 헤더 텍스트만 세로로 표시된다. 접힌 상태에서 해당 열의 셀을 포커스하면 자동으로 펼쳐진다. 사용자가 내비게이션하면서 자연스럽게 열린다.

`act-col-collapsed` 클래스로 접힘 너비를 정의했다. 초기 구현은 `:style="{ width: ... }"`로 인라인 스타일을 썼는데, 이것을 전부 CSS 클래스로 교체했다. 인라인 스타일은 테마별 오버라이드가 어렵고 우선순위가 높아 다른 스타일과 충돌하기 쉽다.

## 정리

RecordSection은 기능 하나씩 붙이면서 복잡해진 컴포넌트다. 각 결정이 독립적으로는 합리적이지만, 쌓이면서 고려해야 할 것들이 늘어났다.

| 문제 | 해결 |
|------|------|
| 바이트 반복 계산 | `TextEncoder` 싱글턴 |
| Map 전체 재렌더 | `ref(Map)` → `reactive(Map)` |
| sticky 열 투명 | 명시적 불투명 배경색 |
| 열 높이 불일치 | `syncRowHeights` 측정 후 최댓값으로 통일 |
| sticky left 오프셋 | 이전 열 너비 누적 합으로 동적 계산 |
| 저장 피드백 | 성공은 조용히(배경색), 실패는 명확하게(빨간 테두리) |
| 인라인 스타일 | CSS 클래스로 이동 — 테마 오버라이드 가능 |
