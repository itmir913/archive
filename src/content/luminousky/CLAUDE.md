# luminousky Chapter 2 포스트 작성 지침

이 CLAUDE.md는 `src/content/luminousky/` 디렉토리 전용이다.
루트의 `CLAUDE.md`와 함께 읽는다 — 이 파일은 루트 지침을 **대체**하지 않고 **보완**한다.

---

## 포스트가 존재하는 이유

Chapter 2(luminousky)는 Chapter 1(ITMir)의 보존 아카이브와 다르다. 과거를 그대로 담는 것이 아니라 **현재 진행 중인 작업의 사고 과정을 기록**한다.

글의 목적은 두 가지다:

1. **내가 왜 이 결정을 내렸는가** — 나중에 같은 문제를 다시 마주쳤을 때 참고할 수 있도록
2. **이 결정이 어떤 맥락에서 나왔는가** — 그 맥락이 사라졌을 때도 결정의 이유를 복원할 수 있도록

독자는 1년 후의 나 자신으로 가정한다. 기술적 배경은 충분하지만 당시의 세부 맥락은 잊어버린 상태로 읽는다고 가정한다.

---

## 파일 구조와 명명 규칙

### 디렉토리 구조

```
src/content/luminousky/
  {project-slug}/
    {post-slug}/
      index.md
      images/           ← 이미지가 있을 때만 생성
```

### Slug 규칙

- 소문자, 하이픈 구분, ASCII만 허용
- 한국어 제목 → 영어 번역 (로마자 변환 금지)
- 디렉토리 이름 = URL slug (Astro가 폴더명으로 라우팅)
- `{post-slug}`는 주제를 명확히 담아야 한다: `db-schema-design-integrity-check`처럼 읽으면 내용이 짐작되는 수준

### frontmatter 스키마

**`src/content/luminousky/template/index.md`** 파일을 복사해서 시작한다.

필드 주의사항:
- 프로젝트 표시 이름은 `src/data/luminousky-projects.ts`에서 slug 기반으로 중앙 관리한다
- `draft: true`로 시작, 완성되면 `draft: false`로 전환
- `date`는 작성 시점 기준 (작업을 마친 날이 아니라 글을 쓰는 날)
- `description`은 한 줄 — 포스트 목록에서 카드 요약으로 표시된다

## 글의 필수 구성 요소

모든 포스트는 아래 요소를 명시적으로 담아야 한다. 소제목 이름이 동일할 필요는 없지만, 내용은 반드시 포함되어야 한다.

### 1. 맥락 (Context / 배경)

이 글이 다루는 작업이 왜 필요했는가. 어떤 문제가 있었고, 그 문제가 왜 문제였는가.

> "이 도구가 이미 있는데 왜 또 만들었나?"라는 질문에 답해야 한다.

### 2. 고려한 대안 (Alternatives Considered)

선택하지 않은 방법들, 그리고 선택하지 않은 이유. 이 섹션이 없으면 독자는 "왜 X를 안 했나?"를 의심하게 된다.

> 대안이 없었다면 그 이유를 적는다. 대안 섹션을 생략하지는 않는다.

### 3. 결정 (Decision)

무엇을 선택했고, 어떤 기준으로 선택했는가. 기술 스택 선택이라면 선택 기준이 명확해야 한다.

> "좋아서" 또는 "익숙해서"는 기록이 아니다. 구체적인 이유가 있어야 한다.

### 4. 결과 또는 현재 상태

결정을 실행한 결과. 예상대로 됐는가, 아니라면 무엇이 달랐는가. 글을 쓰는 시점의 상태를 명확히 한다.

---

## 무엇을 기록해야 하는가

### 기록해야 할 것

- **선택하지 않은 대안과 그 이유** — 나중에 같은 대안을 다시 검토하게 될 때 가장 유용한 정보다
- **당시에는 당연했지만 나중엔 잊어버릴 제약 조건** — "배포 후 스키마 변경이 어렵기 때문에" 같은 맥락
- **예상과 달랐던 결과** — 계획대로 됐다면 짧게, 달랐다면 무엇이 왜 달랐는지
- **다음 포스트로 넘기는 내용** — 이 글의 범위를 명확히 하고 연결을 예고한다

### 기록하지 않아도 되는 것

- 코드 자체의 동작 설명 — 코드를 읽으면 알 수 있는 것은 글로 쓰지 않는다
- 구현 세부사항의 나열 — "A를 했고, B를 했고, C를 했다"는 변경 이력이지 기록이 아니다
- 결국 바꾼 결정 — 번복한 결정은 현재 상태 기준으로 기록한다 (왜 번복했는지는 적어도 좋다)

---

## 글의 범위와 분량

### 한 포스트의 범위

한 포스트는 하나의 결정 단위를 다룬다. 설계 결정, 기술 선택, 릴리즈 전 점검처럼 "왜 이렇게 했나"라는 질문 하나에 대한 답이 되어야 한다.

범위가 너무 크면 나눈다. "개요와 기술 스택"처럼 포스트 제목이 2개를 묶고 있다면 그 자체가 분리 신호다.

### 분량

분량 목표는 없다. 결정과 맥락이 충분히 담겼으면 짧아도 된다. 다만:

- 대안 섹션을 통째로 생략하면 짧아진 것이 아니라 빠진 것이다
- 코드 블록으로 분량을 채우지 않는다

## 언어 선택

- 한국어와 영어를 섞어 쓴다 (기술 용어는 영어 원문 유지)
- 언어 선택 원칙: 해당 포스트의 주된 독자가 한국어 독자인지 영어 독자인지에 따라 결정
  - 교육·학교 관련 도구, 국내 맥락이 강한 주제 → 한국어 주도
  - 기술 선택·아키텍처·오픈소스 도구 → 영어도 자연스럽다
- 한 포스트 안에서 언어를 일관되게 유지한다 — 섹션마다 언어가 바뀌면 안 된다

## 이미지와 첨부파일

### 이미지

`images/` 디렉토리를 만들어 포스트 폴더 안에 둔다.

```markdown
![설명](./images/파일명.jpg)
```

- 스크린샷보다 다이어그램이 먼저다 — 스크린샷은 UI가 달라지면 내용이 틀리지만 다이어그램은 설계를 담는다
- 이미지에는 반드시 alt 텍스트를 작성한다

### 첨부파일

GitHub Releases에 업로드한 뒤 URL로 참조한다.

```
https://github.com/itmir913/archive/releases/download/itmir-attachments/{filename}
```

파일명은 ASCII-safe: 공백 → 하이픈, 한글 금지.

## 새 프로젝트 추가 시 체크리스트

1. `src/data/luminousky-projects.ts`에 `'slug': '표시 이름'` 한 줄 추가
2. `src/content/luminousky/{project-slug}/` 디렉토리 생성
3. `src/content/luminousky/template/index.md` 복사 → frontmatter 수정
4. 첫 포스트 slug는 `{project-slug}-overview` 패턴을 권장 (전체 개요를 먼저 쓰면 이후 포스트가 맥락을 전제할 수 있다)

## AI가 포스트를 작성할 때

AI가 초안을 작성하는 경우 다음을 준수한다:

- **없는 대안을 지어내지 않는다** — 실제로 검토하지 않은 옵션을 "고려한 대안"으로 쓰지 않는다
- **결과는 확인된 사실만** — "이렇게 될 것이다"가 아니라 "이렇게 됐다"
- **날짜는 사용자가 확인** — `date` 필드는 사용자가 실제 작성일로 수정한다
- **draft: true로 남긴다** — AI 초안은 반드시 사용자 검토 후 `draft: false`로 전환
- 글의 1인칭은 저자(luminousky)다 — AI의 관점이나 서술 방식을 따르지 않는다
