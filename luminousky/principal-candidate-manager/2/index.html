학교장추천자 선발 관리 시스템 — Float-Free 아키텍처와 핵심 설계 결정

April 20, 2026 · Principal Candidate Manager

부동소수점 오차를 원천 차단하는 ×100000 정수 아키텍처와 추천 확정의 불변 스냅샷 설계

April 20, 2026 Principal Candidate Manager

Float-Free 아키텍처

학교장추천전형 점수는 소수점이 포함된다. 내신 1.25등급, 봉사 30.5시간 같은 값들이다. 부동소수점 연산의 오차가 점수 계산에 개입하면 추천 결과가 달라질 수 있다.

해결책: 모든 점수를 ×100000 정수로 저장한다.

내신 1.25등급  → 125000
봉사 30.5시간 → 3050000

소수점 5자리까지 오차 없이 표현 가능하고, DB와 Rust 내부는 완전한 Float-Free Zone이 된다.

투명화 계층

변환은 두 경계에서만 일어난다:

  • 입력: 교사 UI "30.5" → 백엔드 수신 즉시 ×100000 → DB 3050000 저장
  • 출력: DB 3050000 → API 직렬화 시 ÷100000 → Vue 표시 "30.5"

백엔드에서만 변환한다. 프론트엔드는 변환 로직을 갖지 않는다.

Rust에서의 강제

Score newtype을 사용해서 컴파일 시점에 정수 확인을 강제한다. REAL 타입 컬럼 사용을 코드 리뷰에서 차단하는 것만으로는 부족하기 때문이다.

Abandon(포기) 박제 로직

추천 확정 후 학생이 입학을 포기해도 results 테이블의 recommended=1 행은 수정하지 않는다. 불변 스냅샷(Immutable Snapshot) 으로 유지한다.

포기는 applications.abandoned=1만 업데이트한다. 잔여석 계산은 이 스냅샷을 기준으로 실시간에 계산한다:

SELECT u.capacity
    - (SELECT COUNT(*) FROM results r2
       WHERE r2.univ_id = u.id
         AND r2.round_id < :current_round_id
         AND r2.recommended = 1)
  AS remaining_seats
FROM universities u

포기자도 recommended=1로 박제되어 있으므로 잔여석 계산에 자동 반영된다.

Fail-Fast 정책

대입 추천 시스템의 특성상 silent fallback을 전면 금지한다. unwrap_or(0), unwrap_or_default(), unwrap_or_else(|| ...) 패턴은 즉시 Err 반환으로 교체한다.

예상치 못한 상태에서 0점이나 기본값으로 계산을 계속하는 것보다 에러를 즉시 노출하는 것이 안전하다. 추천 결과가 틀리는 것이 시스템 오류보다 훨씬 나쁜 결과다.

허용 목록(feedback_silent_fallback_allowed.md)에 24곳의 예외를 명시하고, 새 예외는 반드시 이 파일을 먼저 업데이트하도록 했다.

DB 트리거로 CLOSED 라운드 보호

CLOSED 상태의 라운드 데이터는 삭제·수정이 불가능해야 한다. 미들웨어 검사만으로는 버그나 직접 DB 접근을 막을 수 없다. SQLite 트리거로 DB 수준에서 차단한다.

유일한 예외: applications.abandoned: 0 → 1 업데이트. 포기 처리만 허용한다.

설계 결정 모음

JWT 시크릿 휘발: 서버 시작 시 OsRng로 32바이트 hex를 새로 생성한다. DB에 영속화하지 않는다. 서버 재시작 시 모든 세션이 무효화되는 것이 의도된 설계다.

Excel 열 이름 기반 파싱: 열 인덱스 기반 파싱을 금지하고 헤더 이름 기반으로만 파싱한다. 열 순서가 달라도 올바르게 처리되고, 열 누락 시 즉시 400 오류를 반환한다.

Import 오류 정책: 오류가 있으면 전체 거부(rollback + 422). 부분 import 없음. 중복은 warning이 아닌 error다. 일부만 들어가면 데이터 일관성이 깨진다.
