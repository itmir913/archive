학교장추천자 선발 관리 시스템 — 개요와 기술 스택

April 15, 2026 · Principal Candidate Manager

April 15, 2026 Principal Candidate Manager principal-candidate-manager

배경

한국 고등학교의 학교장추천전형은 대학별 선발 기준에 따라 학생 점수를 계산하고 추천자를 결정하는 복잡한 과정이다. 엑셀로 관리하면 실수가 생기고, 담임 교사와 관리자 간 데이터 공유가 어렵다. 이 과정을 학내 네트워크 안에서 웹 UI로 처리하는 단일 실행 파일(pcm.exe)을 만들었다.

기술 스택

레이어기술선택 이유
프론트엔드Vue 3 + Vite + Tailwind CSS v4SPA, @tailwindcss/vite
백엔드Rust + Axum 0.7 + SQLx 0.8타입 안전, 단일 바이너리
DBSQLite WAL단일 파일, 트랜잭션 안전
배포rust-embed프론트엔드를 바이너리에 임베드, 단일 .exe
인증JWT + bcrypt관리자/담임 역할 분리

dev 빌드는 cargo run --features dev + Vite dev server를 concurrently로 실행한다. 프로덕션에서는 rust-embed로 Vite 빌드 결과물이 바이너리에 포함된다.

주요 기능

관리자 워크플로우

  1. 학급·학생·영역·대학 기초 설정 (Excel 업로드)
  2. 라운드 열기 (OPEN → CLOSED → FINALIZED)
  3. 점수 계산 실행
  4. 결과 확인 및 추천 확정
  5. Excel 내보내기

담임 워크플로우

  1. 라운드 OPEN 시 teacher_editable 전형요소의 학생별 값 웹 입력
  2. 지원자 등록 (학생 + 대학 선택)
  3. 자신의 학급 결과 조회

역할 분리

역할접근 범위
관리자전체 데이터 조회·수정, 점수 계산, 추천 확정
담임자기 학급 학생의 값 입력, 지원 등록, 결과 조회

담임은 JWT 토큰에 grade/class_no가 포함되어 자동으로 자기 학급만 필터링된다.

점수 계산 설계

점수 계산은 반드시 Rust 백엔드에서만 수행한다. 프론트엔드는 score-preview API를 호출해 표시만 한다. 학교장 추천 결과가 실제 입시에 영향을 미치므로 프론트엔드 계산은 법적·신뢰성 리스크가 있다.

영역 유형:

  • NUMERIC: 내신 등급 등 구간 환산표(range_table) 적용
  • CATEGORY: 수상 실적 등 범주 매핑(category_map) 적용
  • MANUAL: 봉사 시간 등 직접 값 입력, 최댓값 제한

상태 전이

라운드: OPEN ↔ CLOSED → FINALIZED

CLOSED 라운드의 데이터는 DB 트리거로 삭제·수정이 차단된다. abandoned=1 업데이트만 예외 허용된다.

테스트

259개 단위·통합 테스트가 전부 통과한다. tests/ 디렉토리 기반으로 구성하고 src/lib.rs로 bin+lib 크레이트 구조를 만들어 외부 테스트에서 내부 함수를 임포트할 수 있다.
