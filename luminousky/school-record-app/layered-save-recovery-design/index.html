저장과 복구의 계층 설계 — 4중 안전망과 외부 복구 원칙

July 9, 2026 · School Record Editor

자동저장·셀 히스토리·스냅샷·백업 파일이 각자 다른 역할을 맡는 구조, DB 복구를 앱 외부에 맡긴 이유

July 9, 2026 School Record Editor school-record-app

왜 저장이 복잡해야 하나

학기말 생기부 마감 상황에서 데이터를 잃는 것은 재앙이다. 교사 입장에서 “방금 수정한 내용이 사라졌다”와 “열흘 전 상태로 돌아가고 싶다”는 완전히 다른 상황이다. 그 두 상황을 같은 기능으로 해결할 수 없다.

이 프로젝트는 저장과 복구를 네 개의 독립된 레이어로 구성했다. 각 레이어는 서로 다른 실패 상황을 다룬다.

4개 레이어와 역할

1. 자동저장 (Auto-save)

타이핑 1초 후 자동으로 SQLite에 기록된다. 저장 버튼이 없다.

이 레이어가 해결하는 것은 유실 방지다. 앱이 갑자기 닫히거나 PC가 꺼져도 마지막 타이핑에서 1초 이내의 내용만 잃는다.

저장 버튼을 없앤 부수 효과가 있다. “저장했나?”를 생각하지 않아도 된다. 교사가 내용에만 집중할 수 있는 심리 상태를 만든다. 기능이 아닌 심리를 설계한 결정이다. 이 결정사항은 프로그램 배포 후 가장 인기 있는 기능 중 하나가 되었다.

2. 셀 단위 히스토리 (Cell History)

특정 셀을 명시적으로 저장하면 이전 버전들이 누적된다. 셀마다 독립된 히스토리가 있다.

이 레이어가 해결하는 것은 단일 셀 단위의 “이전 문장 복구”다. “어제 작성했던 더 좋은 문장이 있었는데”를 다시 꺼낼 수 있다. 스냅샷보다 세밀하고, 자동저장보다 의도적이다.

엑셀로 작업할 때 “최종_진짜최종_수정본2.xlsx”를 만들어가던 패턴을 시스템이 대신한다.

3. 타임머신 스냅샷 (Snapshot)

4중 저장 구조에서 가장 핵심적인 레이어다. “오늘 대규모 수정을 시작하기 전 전체 상태로 돌아가고 싶다”를 해결하는 것이 목표이며, 이 도구의 버전 관리 기능 자체라고 봐도 된다.

파일 복사 방식의 문제

가장 단순한 구현은 스냅샷을 찍을 때마다 .db 파일을 통째로 복사해 보관하는 것이다. 직관적이지만 문제가 있다.

  • 공간: 스냅샷 N개 = DB 크기 × N. 파일 하나가 수 MB라면 스냅샷 10개만으로 수십 MB가 늘어난다.
  • 속도: 찍을 때마다 파일 전체를 디스크에 쓴다. 기록이 많아질수록 점점 느려진다.
  • 복원: 파일을 교체하는 방식이라 복원 중 실패하면 현재 상태까지 날아갈 수 있다.

이 구현은 DB 파일을 전혀 복사하지 않는다.

Git에서 가져온 아이디어, 선형 히스토리로 재설계

Git은 커밋을 “변경된 파일 전체의 스냅샷”으로 저장하면서도 공간을 낭비하지 않는다. 핵심은 콘텐츠 주소 기반 객체 저장소 — 같은 내용이면 같은 해시, 다른 커밋이어도 같은 blob 객체를 가리킨다. 브랜치와 머지를 지원하기 위해 DAG(유향 비순환 그래프) 구조를 쓴다.

생기부 편집에는 브랜치가 없다. 수정은 항상 선형으로 이루어진다. DAG가 필요 없다는 뜻이다. 이 제약 덕분에 구조를 크게 단순화할 수 있었다.

  • Git 커밋 ≈ Snapshot 행 (시점을 가리키는 포인터)
  • Git blob ≈ ActivityRecordHistory 행 (특정 시점의 셀 내용)
  • Git tree 조립 ≈ 타임스탬프 범위 쿼리 (changed_at <= snapshot_at)
  • 브랜치 없음 → 단일 타임라인 → 쿼리로 충분

Snapshot 테이블 자체는 타임스탬프와 메모만 저장한다. 실제 데이터는 ActivityRecordHistory에 있다.

Snapshot 테이블
┌────┬──────────┬─────────────────────┐
│ id │ memo     │ created_at          │
├────┼──────────┼─────────────────────┤
│  1 │ "수정 전"│ 2026-06-01 09:00:00 │
│  2 │ NULL     │ 2026-06-03 14:30:00 │
└────┴──────────┴─────────────────────┘

생성: 변경된 셀만 히스토리에 추가

스냅샷을 찍으면 두 단계가 순서대로 실행된다.

  1. 현재 ActivityRecord 전체를 ActivityRecordHistory에 밀어 넣는다. ActivityRecord.updated_at을 changed_at으로 사용한다. 단, 같은 (activity_record_id, changed_at) 조합이 이미 존재하면 삽입하지 않는다.
INSERT INTO ActivityRecordHistory (activity_record_id, content, changed_at, note)
SELECT r.id, r.content, r.updated_at, NULL
FROM ActivityRecord r
WHERE NOT EXISTS (
    SELECT 1 FROM ActivityRecordHistory h
    WHERE h.activity_record_id = r.id
      AND h.changed_at = r.updated_at
)
  1. Snapshot 테이블에 datetime('now')를 created_at으로 행을 삽입한다.

중복 제거 조건(NOT EXISTS)이 Git의 콘텐츠 주소 방식과 같은 역할을 한다. 두 스냅샷 사이에 수정되지 않은 셀은 히스토리 행이 새로 생기지 않는다. 변경이 전혀 없었다면 저장 비용이 0이다. 스냅샷을 10개 찍어도, 그 사이에 수정한 셀만큼만 히스토리가 늘어난다.

이 특성이 사용자 행동을 바꾼다. 스냅샷을 “아낄” 이유가 없어진다. 시간이 걸리지도 않고, 디스크를 낭비하지도 않으므로 교사가 심리적 부담 없이 언제든 찍을 수 있다. 설계가 올바를 때 사용자는 도구를 사용하는 방식 자체를 바꾼다.

복원: 타임스탬프 기준 시간 역 쿼리

복원은 Snapshot.created_at을 기준으로 각 셀의 “그 시점 직전 최신 버전”을 찾아 덮어쓰는 것이다.

UPDATE ActivityRecord SET
  content = COALESCE(
    (SELECT h.content
     FROM ActivityRecordHistory h
     WHERE h.activity_record_id = ActivityRecord.id
       AND h.changed_at <= :snapshot_at     -- 스냅샷 시점 이전만
     ORDER BY h.changed_at DESC LIMIT 1),   -- 그 중 가장 최근
    ''                                      -- 해당 시점 이전 히스토리 없으면 빈 문자열
  )

스냅샷 시점 이후에만 히스토리가 존재하는 셀 — 스냅샷을 찍은 뒤에 새로 생성된 셀 — 은 서브쿼리가 NULL을 반환하고 COALESCE에 의해 ''(빈 문자열)로 초기화된다. “스냅샷 당시에는 이 셀이 없었다”는 사실을 빈 문자열로 표현하는 것이다. 별도로 삭제하거나 마킹하지 않아도 되고, 쿼리 하나로 “시점 이전 최신값 또는 빈 문자열”을 일관되게 처리한다.

공간 효율 정리

방식스냅샷 N개의 저장 비용
파일 복사O(N × DB 크기)
이 구현O(스냅샷 사이 변경된 셀 수 합계)

변경이 없으면 비용이 없다. 변경이 많아도 변경된 셀의 텍스트 크기만큼만 늘어난다. 파일 전체를 박제하는 방식과 달리 불변 부분을 중복 저장하지 않는다.

4. .db.backup 자동 생성

앱이 DB를 열 때마다 .db 파일을 복사해 .db.backup으로 저장한다. 최후의 방어선이다.

이 레이어가 해결하는 것은 DB 파일 자체의 손상이다. 위 세 레이어는 모두 DB가 정상적으로 열릴 수 있다는 것을 전제한다. DB 파일이 손상되면 앱 자체가 열리지 않는다. 이때 .db.backup 파일을 .db로 이름을 바꿔서 복구한다.

레이어 간 역할 비중복

중요한 것은 네 레이어가 역할이 겹치지 않는다는 것이다.

레이어복구 대상복구 단위발동 조건
자동저장유실실시간항상 자동
셀 히스토리수정 취소셀 하나명시적 저장 시
스냅샷전체 롤백DB 전체사용자 요청
.db.backupDB 손상DB 파일앱 실행 시 자동

자동저장이 있다고 스냅샷이 필요 없는 것이 아니다. 스냅샷이 있다고 .db.backup이 필요 없는 것도 아니다. 각자 다른 실패 시나리오를 다룬다.

이 구조를 설계할 때 “어떤 기능을 추가할까”가 아니라 “어떤 상황에서 교사가 데이터를 잃는가”를 먼저 나열했다. 상황별로 레이어를 매핑했다.

DB 복구를 앱 외부로 설계한 이유

.db.backup 복구를 앱 내부 UI로 제공하지 않았다. 사용자는 파일 탐색기에서 파일 이름을 직접 바꿔야 한다.

처음에는 이것이 아쉬운 점처럼 보인다. 교사 대상 도구에서 파일 탐색기 조작을 요구하는 것은 진입 장벽이다.

그런데 다른 방향에서 보면 이 결정이 맞다.

DB 파일이 손상됐다는 것은 앱이 그 파일을 열 수 없다는 것이다. 앱이 파일을 열지 못하는 상황에서 앱 UI를 통해 복구를 시도하면, 앱이 손상된 파일을 열려다 실패하면서 그 과정에 백업 파일까지 덮어쓰는 최악의 시나리오가 생긴다.

손상된 도구로 자기 자신을 고치면 안 된다. 복구 작업은 도구 밖에서 이루어져야 한다. 불편해 보이지만 더 안전한 방향이다.

같은 원칙이 다른 곳에도 적용된다. 백업 파일을 앱 내부 저장소가 아니라 사용자가 직접 접근할 수 있는 같은 폴더에 두는 것도 마찬가지다. 앱이 완전히 제거되거나 동작하지 않아도 데이터는 살아남는다.

GitHub Actions → Releases → 랜딩 페이지 배포

저장과 직접적인 관련은 없지만, 같은 “장기 지속 가능성” 관점에서 배포 구조도 짚을 만하다.

배포 인프라 비용은 0원이다.

코드 푸시 (태그)
    ↓
GitHub Actions (Windows + macOS 크로스 빌드)
    ↓
GitHub Releases (빌드 결과물 자동 업로드)
    ↓
랜딩 페이지 (GitHub API → 다운로드 버튼 자동 갱신)

버전이 올라가면 태그를 푸시한다. 나머지는 자동이다. 교사는 GitHub을 거치지 않고 랜딩 페이지의 버튼 하나로 받는다. 버전이 바뀌어도 랜딩 페이지 HTML을 수정하지 않아도 된다.

API 응답 실패 시에는 /releases/latest로 자동 폴백된다. 최악의 경우에도 사용자는 GitHub Releases 페이지로 이동해 직접 받을 수 있다.

포터블(설치 없는) 버전을 별도로 제공하는 것도 학교 환경을 고려한 결정이다. 망분리 PC나 설치 권한이 없는 환경에서 USB에 담아 실행할 수 있다.
