All-in-One 학교생활기록부 에디터 — 개요와 기술 스택 선택

April 1, 2026 · School Record Editor

왜 만들었고, Tauri + Rust + Vue + SQLite를 선택한 이유

April 1, 2026 School Record Editor school-record-app

왜 만들었나

학교에서 생활기록부(생기부)를 작성할 때, 여러 학생의 활동 기록을 일관성 있게 관리하고 내보내는 일이 생각보다 복잡하다. 특히 특정 단어를 일괄 치환하거나, 유의어가 중복되지 않았는지 검사하거나, 글자 수 제한을 지키면서 내용을 정리하는 작업은 반복적이고 오류가 생기기 쉽다.

기존에는 엑셀이나 메모장 수준의 도구를 쓰거나, 나이스(NEIS) 시스템에서 직접 편집하면서 불편함을 감수했다. 이 불편함을 해결하기 위해 오프라인 데스크탑 앱으로 만들기로 했다.

핵심 기능 목표

  • 학생·영역·활동 관리: 프로젝트 단위로 학생 목록과 평가 영역/활동을 구성
  • 기록 편집: 학생별 활동 기록을 셀 단위로 편집, 이력 보관
  • 치환 규칙: 정규식 포함 일괄 치환, 기본 규칙 내장
  • 유의어 검사: 유의어 그룹 관리, 원본 텍스트에서 유의어 위치 하이라이트
  • 내보내기: 글자 수 제한 확인, 면책 안내 포함 출력
  • 스냅샷: 작업 중간 상태 저장 및 복원
  • 암호화: 학생 이름·기록 등 민감 데이터 선택적 암호화

기술 스택 선택

Tauri (Rust 백엔드 + 웹프론트엔드)

Electron 대신 Tauri를 선택한 이유는 하나다: 번들 크기와 메모리 사용량. Electron은 Chromium과 Node.js를 통째로 들고 다니는 반면, Tauri는 OS의 WebView를 사용하고 백엔드만 Rust로 컴파일한다. 교사 업무용 도구인 만큼 학교 PC에 부담 없이 설치될 수 있어야 했다.

Rust

백엔드 언어로 Rust를 선택한 것은 타입 안전성과 테스트 편의성 때문이다. 치환 엔진처럼 순수 함수로 추출할 수 있는 로직은 단위 테스트로 완전히 검증하고 싶었다. Rust의 cargo test는 이 목적에 이상적이다.

Vue 3 + Tailwind CSS

프론트엔드는 Vue 3로 구성했다. 섹션(Section)과 모달(Modal) 구분이 명확하고, Pinia 스토어를 통해 백엔드 invoke 호출을 캡슐화하는 구조를 잡았다. 컴포넌트가 Tauri invoke를 직접 호출하지 않고 반드시 스토어를 경유하는 것이 원칙이다.

SQLite

프로젝트 파일 형식으로 SQLite를 선택했다. 단일 .db 파일로 관리되므로 이동과 백업이 간단하고, 트랜잭션 지원으로 데이터 무결성을 보장할 수 있다. 나중에 암호화를 도입할 때도 DB 전체 암호화(SQLCipher) 대신 필드 단위 암호화를 선택하게 되는 결정적 요인이 된다.

앱 구조 개요

프로젝트 파일 (.db)
├── Student       학생 목록
├── Area          평가 영역
├── Activity      활동 항목
├── ActivityRecord 기록 (학생 × 활동)
├── ActivityRecordHistory 기록 이력
├── Snapshot      스냅샷
├── ReplaceRule   치환 규칙
├── SynonymGroup  유의어 그룹
├── SynonymItem   유의어 단어
└── APP_CONFIGS   앱 설정 (암호화 salt 등)

프론트엔드는 섹션 단위로 분리된다: 학생(Student), 영역(Area), 활동(Activity), 기록(Record), 검사(Inspect), 치환(Replace), 내보내기(Export), 체크리스트(Checklist), 설정(Settings), 개요(Overview).
