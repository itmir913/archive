Docker Compose와 정적 JudgeHost — 동적 스케일링을 폐기한 이유

July 17, 2026 · GFPC Programming Challenge

재부팅 후에도 사람이 손대지 않아도 되는 채점 서버를 만들기까지의 시행착오

July 17, 2026 GFPC Programming Challenge gfpc

맥락

리눅스 서버 구축 경험도 있고, DOMjudge를 한 번 설치해본 적도 있었다. 그런데 실제로 GFPC용 서버를 만들면서 가장 어려웠던 두 가지는 따로 있었다.

  1. DOMjudge를 Docker로 어떻게 돌릴 것인가
  2. JudgeHost(채점기)를 여러 개 돌려서 병렬 채점을 어떻게 구성할 것인가

서버 스펙은 RAM 23GB, CPU 16 논리 스레드(인텔, 멀티스레딩 우수)였다. 실측해보니 평상시 부하가 5%도 안 될 만큼 여유가 있었다. 프로그램 채점은 CPU 단일 코어 성능에 더 큰 영향을 받기 때문에, JudgeHost 개수를 CPU 코어(논리 스레드) 개수에 맞춰 할당한다는 방향은 처음부터 명확했다.

고려한 대안

네이티브 설치

정보 교사 카페에 올라온 과거 사례들을 보면 대부분 리눅스 스크립트로 DOMjudge를 직접 설치하는 방식이었다. 어떤 선배 교사는 계정 생성 같은 작업을 데이터베이스에 SQL을 직접 입력해서 처리하기도 했다.

네이티브 설치는 재설치가 번거롭고, 나 아닌 다른 사람이 같은 환경을 다시 만들기 어렵다는 문제가 있었다. 그래서 처음부터 Docker를 쓰겠다고 마음먹었다.

서버를 역할별로 분리

웹 서버, 채점 서버, 디스플레이 서버를 각각 다른 컴퓨터로 나눠 운영하는 사례도 많았다. 하지만 우리에게는 그렇게 여러 대를 굴릴 만한 서버 여유가 없었다. 한 대로 웹 서버, DB, JudgeHost를 전부 감당해야 했다.

JudgeHost 동적 스케일링

처음에는 docker scale 명령으로 JudgeHost 개수를 동적으로 늘리는 방식을 시도했다. CPU 코어 개수만큼 JudgeHost를 할당하면 병렬 채점이 가능해진다는 아이디어 자체는 맞았다.

문제는 재부팅이었다. 서버를 껐다가 켜면 scale로 늘려놓은 JudgeHost 개수가 원래대로 돌아가 버렸다. 그러면 재부팅 후 JudgeHost가 하나밖에 남지 않는다.

동적으로 CPU 코어 개수에 맞춰 JudgeHost를 늘린다는 발상 자체도, CPU 코어 수는 서버를 바꾸지 않는 한 고정값이기 때문에 애초에 “동적으로 늘려야 할” 이유가 없었다.

결정

JudgeHost 개수는 Docker Compose 파일에 정적으로 명시했다. CPU 논리 스레드 개수에 맞춰 JudgeHost 컨테이너를 필요한 만큼 정의해두고, ID만 다르게 해서 복사-붙여넣기 하는 방식이 가장 간단했다. 예를 들어 논리 스레드가 32개면 0번부터 31번까지 정의해두고, 스레드가 5개뿐인 환경이면 다섯 개만 남기고 나머지를 지우면 되는 구조다.

이 결정의 기준은 단 하나였다. 재부팅 후에도 서버 관리자가 아무 명령어도 입력하지 않아야 한다. 서버를 껐다가 다시 켜는 상황(예: 서버실 이전 설치)은 반드시 생긴다. 그때 관리자가 로그인해서 뭔가를 입력해야 한다면, 그건 대회 당일 100% 누락될 거라고 판단했다. AI가 “이 정도 수준의 수동 명령이면 타협 가능한가요?”라고 물었을 때도 나는 타협하지 않았다.

Docker Compose 파일 자체도 많이 참고할 자료가 없었다. DOMjudge 9.x 버전을 Docker로 올리는 방법에 대한 매뉴얼이 거의 없어서, 직접 시행착오를 거쳐 만드는 수밖에 없었다. Compose 파일을 완성하는 데 거의 한 달이 걸렸다.

결과

  • 재부팅해도 JudgeHost 개수가 그대로 유지되는 정적 Compose 구조를 완성했다.
  • 행사 당일 CPU 부하는 코어별로 50%를 넘는 경우가 없었고, 평균 10% 수준이었다. 지금 규모의 3배 정도까지는 여유가 있다고 판단했다(다만 RAM이 32GB였다면 더 여유로웠을 것이다).
  • 하나의 docker compose up 명령으로 웹 서버, DB, 여러 개의 JudgeHost가 한 번에 올라오는 구조이기 때문에, 다른 학교나 다른 교육청도 이 Compose 파일만 있으면 같은 환경을 재현할 수 있다.

지금 다시 처음부터 만든다면, 서버 구축 자체에는 시간을 덜 쓸 것 같다. Compose 파일이 이미 완성되어 있어서, 리눅스 서버에 DOMjudge를 올리는 것 자체는 예전만큼 오래 걸리지 않을 것 같다. 대신 문제 패키징과 테스트케이스 검증에 훨씬 더 많은 시간을 쏟을 것 같다.
